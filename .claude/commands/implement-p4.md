# P4実装コマンド: エクスポート一式 + PWA化 + 非機能要件検証(リリース判定)

## 1. あなたの役割と絶対条件

P1(`implement-p1.md`)・P2(`implement-p2.md`)・P3(`implement-p3.md`)と同一の絶対条件を引き継ぐ。

1. **仕様書がSingle Source of Truthである。** 対象仕様書: `$ARGUMENTS`(未指定時は `docs/trade-journal-spec-v0.5.md`)。特に**3.2節(データ消失リスク対策)・4.2〜4.4節(F1〜F3のエクスポート/バックアップ関連記述)・7章(エクスポート仕様)・8章(非機能要件)**を精読すること
2. 本コマンドはP4スコープのみを実装する。P4は仕様書10章の最終フェーズ(リリース判定ゲート)であり、これ以降の新規フェーズは計画されていない
3. 仕様と本コマンドの矛盾・解釈の余地がある場合は、実装せず質問して停止する
4. 各フェーズ末尾の停止ポイントで停止し、レビュー承認を待つ
5. **P1・P2・P3の成果物を壊さない。** 既存のドメイン型・repository・画面A〜Eの受入基準はP4完了後も全項目維持されること(回帰確認を最終フェーズで行う)

## 2. 前提条件(F0で確認)

- P1(F0〜F4)・P2(F0〜F6)・P3(F0〜F7)が完了し、受入基準を満たしたコードベースが存在する
- Dexieスキーマは version 4(securities / trades / rules / ruleVersions / tradeRuleLinks / tradeMatches / journalEntries / tags / journalTags / priceSnapshots / importBatches / sectors / fxRates / targetAllocations / nisaUsages / cashBalances / appMeta)
- **JSONバックアップ・リストア機能は既にP2完了後に前倒し実装済み**(`domain/backup.ts`・`db/repository.ts`の`restoreFromBackup`、UIは`app/Header.tsx`の「バックアップ(JSON)」「復元(JSON)」ボタン)。本コマンドはこの前倒し実装済み範囲を重複実装せず、**最終バックアップ日時の記録・表示・7日超過警告**のみを追加する
- `navigator.storage.persist()`呼び出し・結果のappMeta記録はP1 F1で実装済み(`db/init.ts`)。本コマンドの対象外
- CLAUDE.md のスコープ記述が「P4実装を許可」に更新されていること(未更新なら停止して報告)

## 3. P4スコープ定義

### 3.1 実装する(In)

CLAUDE.md記載の「P4未着手スコープ」と仕様書7章・8章に対応する以下一式:

- **Excel/CSVエクスポート**: 取引一覧・ジャーナル・保有一覧をBOM付きUTF-8 CSVで出力(仕様書7章「Excel/CSVエクスポート」。既存`domain/sectorCsv.ts`のCSV生成パターンを踏襲し、SheetJS等のバイナリxlsx生成ライブラリは導入しない — F0で異なる方針が必要と判明した場合は停止して質問する)
- **ルールのMarkdownエクスポート**: ルール1件を1ファイルのMarkdownとして出力。ファイル名規約 `rule-{slug}-v{version}.md`(仕様書4.3節)
- **PWA化**: Service Workerによるオフラインキャッシュ、Web App Manifestによるホーム画面追加対応。オフラインでも閲覧・入力可能とする(仕様書8章)
- **最終バックアップ日時表示・7日超過警告**: appMetaに最終バックアップ日時を記録し、起動時に表示。7日超過で警告表示(仕様書3.2節)
- **Safari向け注意表示**: Safari利用時にITPによるストレージ削除リスクの注意文言を画面上に表示(仕様書8章)
- **機密性の明記**: サーバーへのデータ送信ゼロ・外部APIコールなしである旨を画面上に明記(仕様書8章)
- **非機能要件の性能検証**: 取引5,000件・ジャーナル3,000件規模のテストデータで一覧表示・全文検索が体感1秒以内であることを検証する(自動テストまたは手動計測。手法はF0で確定する)

### 3.2 実装しない(Out)

- 株価API連携・リアルタイム株価取得(仕様書2.2節「対象外」。現在値更新はポートフォリオCSV再インポートまたは手動入力のみ、P3までの方針を継続)
- JSONバックアップ・リストアのコア機能自体(P2完了後に前倒し実装済み。本コマンドは最終バックアップ日時表示のみ追加)
- P1〜P3で確定した機能への仕様変更(バグ修正を除く)

## 4. データ層仕様(変更点)

- Dexieスキーマの**version自体は4のまま変更なし**(新規テーブル追加は不要)
- appMetaに以下のキーを追加(F0で最終的なキー名・値の型を確定する):
  - 最終バックアップ日時(例: `lastBackupAt`、ISO 8601文字列)。`app/Header.tsx`の`handleBackup`成功時に書き込む
- 上記以外のドメイン型・テーブル追加は想定しない(F0で追加要件が判明した場合は停止して質問する)

## 5. エクスポート実装要件

### 5.1 取引一覧・ジャーナル・保有一覧CSVエクスポート

- 出力形式: BOM付きUTF-8 CSV(`domain/sectorCsv.ts`の既存パターンを踏襲)
- 列定義はF0で仕様書5章のエンティティ定義と既存一覧画面(`app/trades`・`app/journal`・`app/portfolio`)の表示項目から確定し、要確認事項として列挙・回答を得てから実装する(推測で決定しない)
- 金額は整数(JPY: 円、USD: セント)のまま出力するか、表示用に換算するかをF0で確認する
- エクスポート起点はそれぞれの一覧画面(`/trades`・`/journal`・`/portfolio`)にボタンを追加する

### 5.2 ルールMarkdownエクスポート

- 出力対象: ルール1件(特定バージョンまたは最新バージョン — F0で確認)を1ファイルのMarkdownとして出力
- ファイル名: `rule-{slug}-v{version}.md`。`slug`の生成規則(ルール名からの変換ルール)をF0で確認する
- Markdown本文の構成: 概要・エントリー条件・イグジット条件・除外条件・資金管理(仕様書4.3節のルール構造化セクションに対応)
- エクスポート起点は`app/rules`のルール詳細/一覧画面にボタンを追加する
- 変換ロジックは`domain/`配下の純粋関数とし、ユニットテストを書く

## 6. PWA化実装要件

- `output: 'export'`静的エクスポート構成を維持したままPWA化する(サーバーサイド処理禁止の制約は不変)
- Web App Manifest(`public/manifest.json`または同等)を追加し、アイコン・アプリ名・テーマカラー等を設定する
- Service Workerを追加し、静的アセット(HTML/CSS/JS)をキャッシュしてオフライン起動・閲覧を可能にする。IndexedDBデータ自体はService Workerの対象外(ブラウザのIndexedDBがそのまま機能する)
- キャッシュ戦略・Service Worker実装手段(手書き vs `next-pwa`等のライブラリ)はF0で確認する(新規依存追加の要否を含む)
- オフライン時の挙動(データ書き込み可否、CSVインポート等サーバー通信を伴わない機能は継続利用可能なはず)をF0で整理し、受入基準に反映する

## 7. 最終バックアップ日時・警告・注意表示実装要件

- `app/Header.tsx`の`handleBackup`成功時にappMetaへ最終バックアップ日時を記録する(2章参照)
- アプリ起動時(共通レイアウト、`app/layout.tsx`または`Header.tsx`)に最終バックアップ日時を表示し、7日を超過していれば警告(視覚的に区別できる表示)を出す
- Safari判定(User Agent等。判定方法はF0で確認)時に、ITPによるストレージ削除リスクの注意文言を表示する(仕様書8章)
- 機密性(サーバー送信ゼロ・外部APIコールなし)の明記を画面上の適切な位置(共通レイアウトまたは専用About的な箇所)に追加する

## 8. 非機能要件検証

- 取引5,000件・ジャーナル3,000件規模のテストデータ生成手段を用意する(スクリプトまたはテスト内でのシード生成。実データを使わない)
- 一覧表示・全文検索の体感速度(1秒以内)を検証する。自動テストで閾値を機械的に検証するか、手動計測で確認するかはF0で確認する
- 検証結果は完了報告に記録する(数値・手法を明記)

## 10. 受入基準

**L群(CSVエクスポート)**
- [ ] 取引一覧CSVがBOM付きUTF-8で出力され、Excelで文字化けなく開ける
- [ ] ジャーナルCSVがBOM付きUTF-8で出力され、Excelで文字化けなく開ける
- [ ] 保有一覧CSVがBOM付きUTF-8で出力され、Excelで文字化けなく開ける
- [ ] 出力列がF0で確定した定義と一致する

**M群(ルールMarkdownエクスポート)**
- [ ] ルール1件が`rule-{slug}-v{version}.md`のファイル名でMarkdown出力される
- [ ] 出力内容が概要・エントリー条件・イグジット条件・除外条件・資金管理の全セクションを含む
- [ ] 変換ロジック(純粋関数)にユニットテストがある

**N群(バックアップ日時・注意表示)**
- [ ] バックアップ成功時に最終バックアップ日時がappMetaに記録される
- [ ] 起動時に最終バックアップ日時が表示され、7日超過で警告が出る
- [ ] Safari利用時にITPリスクの注意文言が表示される
- [ ] サーバー送信ゼロ・外部APIコールなしの明記が画面上にある

**O群(PWA化)**
- [ ] Web App Manifestが存在し、ホーム画面に追加できる
- [ ] Service Workerがインストールされ、オフラインでアプリシェルが表示される
- [ ] オフライン時もIndexedDBデータの閲覧・入力が可能である
- [ ] `output: 'export'`静的エクスポート構成が維持されている(`npm run build`が成功する)

**P群(非機能要件検証)**
- [ ] 取引5,000件・ジャーナル3,000件規模で一覧表示・全文検索が体感1秒以内であることが検証・記録されている

**共通群**
- [ ] P1・P2・P3受入基準の全項目が引き続き満たされる(回帰確認)
- [ ] Dexieスキーマversionに変更がある場合、P1〜P3データが無損失で移行される(変更しない場合は本項目は該当なしと明記)
- [ ] 金額を浮動小数点で保持している箇所がない(新規追加分含む)
- [ ] `npm run build` が警告なしで成功する
- [ ] 仕様書10章のP4ゲート(リリース判定)に対する結論を完了報告に明記する

## 11. コーディング規約(P1・P2・P3から追加)

- P1・P2・P3規約を踏襲(strict、`any`禁止、純粋関数分離、仕様書対応コメント、`[P4-Fx]`コミット)
- CSV/Markdown変換ロジックはすべて`domain/`配下の純粋関数とし、Dexie・Reactに依存させない
