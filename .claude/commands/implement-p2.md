# P2実装コマンド: SBI CSVインポート + FIFO損益マッチング + ジャーナル

## 1. あなたの役割と絶対条件

P1(`implement-p1.md`)と同一の絶対条件を引き継ぐ。

1. **仕様書がSingle Source of Truthである。** 対象仕様書: `$ARGUMENTS`(未指定時は `docs/trade-journal-spec-v0.4.md`)。特に**第6章(CSV確定仕様)と4.2節(ジャーナル・FIFO)を精読**すること

### 4.1 スキーママイグレーション

```typescript

  trades:       'id, tradeDate, securityId, [securityId+accountType]',
  rules:        'id, status',
  ruleVersions: 'id, ruleId, [ruleId+version]',
  tradeRuleLinks: 'tradeId, ruleVersionId',
  tradeMatches: 'id, sellTradeId, buyTradeId',
  journalEntries: 'id, tradeId, entryDate',

```

- **マイグレーション後、P1で登録済みの全データが無傷で読めること**(F1の受入条件)
- 既存 Security 型にフィールドを追加: `market: string | null`(仕様書5.1の「市場区分」に対応。P1コマンドでは省略していた。CSV取込時に市場列から設定、手動入力・投信は null 可。既存データはマイグレーションで null を設定)。コードあり銘柄の一意性は (code, market) の組、投信は normalizedName で判定する
- 既存 Trade 型に任意フィールドを追加: `source?: { kind: 'manual' | 'csv'; batchId?: string }`(手動入力は従来通り。CSV取込行にはbatchIdを付与し、取込単位のトレーサビリティを確保する)

## 5. CSVインポート実装要件

**列定義・値ドメイン・変換ルールは仕様書6.2〜6.5節を正とする(本コマンドに転記しない。必ず原文を参照して実装すること)。** 以下は実装構造の要件。

### 5.1 共通基盤(`import/` 配下)

- 文字コード自動判定: Shift-JIS / BOM付きUTF-8 / UTF-8(判定ロジックは純粋関数+ユニットテスト)
- ヘッダー行検出: 先頭セルの値(`約定日` / `国内約定日` / ポートフォリオはセクション構造)で**行位置に依存せず**検出。検出失敗時は「どのファイル種別とも一致しない」旨の明示的エラー
- **列マッピング定義は設定オブジェクトとして外部化**(仕様書6.5)。ファイル種別ごとに1モジュール
- 値変換: `--`/空→null、預り列のトリム、全角(U+3000含む)のNFKC正規化キー生成、`20.5100USD` 形式の通貨サフィックスパース
- 語彙マッピング表(コード内に定数として定義): `株式現物買`/`現買`→buy、`株式現物売`/`投信金額解約`/`現売`→sell、`投信金額買付`/`分配金再投資`→buy、` NISA(成) `→nisa_growth、`NISA`(米国)→nisa_growth、` NISA(つ) `→nisa_tsumitate、` 特定 `/`特定`→specific、` 旧NISA `/`旧NISA`→old_nisa
- 銘柄の自動照合: コードあり→code一致、なし(投信)→normalizedName一致。**不一致時は取込を中断せず「未照合リスト」としてUIに提示し、ユーザーが既存銘柄への対応付け or 新規銘柄作成を選択**。対応付けはエイリアスとして保存し次回から自動照合(仕様書6.3)
- 重複判定: 仕様書6.2/6.4の判定キー。**同一キー衝突は自動スキップせず、対象行を提示してユーザーが取込可否を判断**(仕様書6.2)
- インポート結果サマリー表示(取込/スキップ/エラー件数+エラー行番号と理由)、ImportBatchとして永続化

### 5.2 ファイル別

- **約定履歴照会CSV**: 商品種別判定(取引列の値で投信/株式・ETFを分岐)、投信は口数・1万口あたり基準価額として解釈
- **米国株式約定履歴CSV**: USD建て。amountはセント整数化。実効コスト `|数量×単価−受渡金額|` を参考値として算出しTradeに保持(任意フィールド `impliedCost?: number`)
- **ポートフォリオCSV**: セクションタイトルから商品種別・口座区分をパースし、明細のみPriceSnapshot化。合計ブロックは取り込まず明細再計算と突合し、不一致は警告表示。**取引記録由来の保有数量とCSV保有数量の差異レポート**を表示(取込漏れ検出。仕様書6.3)

### 5.3 UI(画面D: インポート `/import`)

- ファイル選択(ドラッグ&ドロップ可)→ 種別自動判定 → プレビュー(先頭20行の解釈結果)→ 未照合銘柄の解決 → 重複衝突の解決 → 取込実行 → 結果サマリー
- 取込履歴一覧(ImportBatch)を表示

## 6. FIFO損益マッチング実装要件

### 6.1 アルゴリズム(`domain/fifo.ts` — 純粋関数)

- **グルーピングキー: securityId + accountType。** 口座区分をまたぐマッチングは行わない(NISAの売却が特定口座の買付を消費することは制度上あり得ないため)。この決定は仕様書に明記がないため、コード先頭コメントに根拠を記載すること
- 同一グループ内で約定日昇順(同日内は createdAt 昇順)に処理。売却1件が複数の買付に跨る部分マッチを許容し、マッチごとに消費数量を記録する
- 実現損益 = 売却の受渡金額按分 − 買付の受渡金額按分。**按分は数量比で行い、端数(円/セント)は各取引の最後のマッチに寄せる**(按分合計が元金額と一致することをテストで保証)
- 買付残が不足する売却(初期投入範囲=直近2年より前の買付に依存するケース)は**未解消売却として警告一覧に表示**。解消手段は「ユーザーが期首残高相当の買付取引を手動登録して再マッチング実行」とし、専用の期首残高エンティティは作らない
- 再マッチング: CSV取込・取引の追加/編集/削除後に、影響を受けたグループのみ `fifo_auto` のマッチを破棄して再計算する。`manual` のマッチは保持し、手動マッチ済み数量を除外してFIFOを適用する

### 6.2 手動修正UI

- 取引詳細から、売却に対するマッチ先買付の変更(候補=同グループ内の残数量がある買付)と数量指定ができる
- 手動マッチは `method: 'manual'` で保存し、自動再計算で消えない

### 6.3 表示

- 取引一覧(画面A)の売却行に実現損益列を追加(通貨付き、正負で色分け)
- 取引詳細にマッチ内訳(どの買付から何株、損益いくら)を表示
- ※ルール別・期間別の損益集計はP3スコープのため実装しない

## 7. ジャーナル実装要件(画面E: `/journal`)

- エントリ作成: 取引に紐付ける(取引一覧・詳細から遷移)か、日付のみ指定(日単位)
- 本文: プレーンテキストの長文(数千字)。自動保存(編集停止2秒後)+明示保存
- タグ: 感情タグ(シード4種+追加可)と自由タグ。エントリに複数付与可。タグマスタ管理(改名・削除。削除時は関連 journalTags も削除)
- 一覧: 日付降順。取引紐付きエントリは銘柄・売買・金額を併記
- 全文検索: 本文+タグ名を対象とした部分一致(NFKC正規化した上で照合)。5,000件規模で体感1秒以内(仕様書8章)。実装は全件ロード+メモリ内フィルタで可(外部検索ライブラリ不要)

## 9. 受入基準

**D群(インポート)**
- [ ] 実ファイル3種(ユーザーが手元で用意)がそれぞれ正しく種別判定・取込できる
- [ ] Shift-JISとUTF-8(BOM有無)の等価ファイルが同一結果になる(ユニットテスト)
- [ ] 同一ファイルの再取込で、重複衝突が提示され「すべてスキップ」を選ぶと0件取込になる
- [ ] 投信(コードなし)の未照合→手動対応付け→エイリアス保存→再取込で自動照合、が通る
- [ ] 米国株のamountがセント整数で保存され、実効コストが導出されている
- [ ] ポートフォリオ取込で合計ブロック突合と保有数量差異レポートが表示される
- [ ] 不正ファイル(ヘッダー不一致)で明示的エラーになり、部分取込が発生しない

**E群(FIFO)**
- [ ] 売却1件が複数買付に跨るケースで数量・損益の按分が正しい(端数含めユニットテストで検証)
- [ ] 口座区分が異なる同一銘柄がマッチングされない
- [ ] 買付不足の売却が未解消警告に表示され、期首相当の買付を手動登録→再マッチングで解消する
- [ ] 手動マッチが自動再計算後も保持される
- [ ] 取引の削除・編集後に該当グループのみ再計算され、他グループのマッチが変化しない
- [ ] 按分合計=元金額の恒等性テストがある(JPY/USD両方)

**F群(ジャーナル)**
- [ ] 取引単位・日単位の両方でエントリを作成でき、数千字の本文が保存・再表示される
- [ ] 感情タグ・自由タグの付与/解除、タグマスタの改名・削除(関連の同時削除)ができる
- [ ] 正規化を考慮した部分一致検索(全角/半角ゆれ)がヒットする

**共通群**
- [ ] P1受入基準の全項目が引き続き満たされる(回帰確認)
- [ ] version 1 → 2 のマイグレーションでP1データが無損失
- [ ] 金額を浮動小数点で保持している箇所がない(実現損益・実効コスト含む)
- [ ] `npm run build` が警告なしで成功する
- [ ] **テスト用CSVフィクスチャは匿名化したダミーデータで作成し、実ファイル(実取引データ)をリポジトリにコミットしない**
