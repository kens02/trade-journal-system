## 1. 目的・背景

個人投資家(SBI証券利用)の取引活動を対象に、以下3機能を統合したWebアプリを構築する。

1. **トレードジャーナル**: 取引記録と振り返り(長文メモ・感情タグ)を蓄積し、成績を可視化する
2. **売買ルール仕様化**: 売買ルールを構造化文書として記述・バージョン管理し、各取引に紐付ける
3. **ポートフォリオ・リバランス計算**: 目標配分と現状の乖離から必要売買数量を算出する

本アプリの独自価値は「取引記録 ⇔ ルール仕様」の往復構造にある。ルールをSingle Source of Truthとして扱い、実際の取引がどのルール(のどのバージョン)に従ったかをトレース可能にする。

## 2. スコープ

### 2.1 対象

- SBI証券における**現物取引**: 日本株、米国株、投資信託、ETF(国内・米国)
- 取引データの取り込みはSBI証券からダウンロードしたCSVファイルのインポートによる(API連携は行わない)
- 初期データ投入は**直近2年以内**の取引履歴を対象とし、必要に応じて追加投入する(C4決定)
- 単一ユーザー・単一ブラウザでの利用

### 2.2 対象外(v1時点)

- 信用取引、FX、先物・オプション、暗号資産
- 証券会社APIによる自動連携、リアルタイム株価取得
- 複数ユーザー・複数端末間のデータ同期
- 税務計算(確定申告用の損益計算書出力等)

## 3. システム構成方針

### 3.1 前提: Vercel無料枠(Hobbyプラン)での恒久運用

| 方針 | 内容 | 理由 |
|---|---|---|
| 静的サイト中心 | Next.js(App Router)を静的エクスポートまたはクライアントレンダリング主体で構成。サーバーサイド処理は原則使わない | 無料枠の関数実行制限・帯域制限の影響を最小化 |
| サーバーDBなし | 永続化はブラウザの**IndexedDB**。サーバー側には一切データを保存しない | 無料DBの容量・休眠制限を回避。取引データという機密情報を外部に置かない |
| バックアップはファイル | 全データをJSON(完全バックアップ)およびExcel/CSV(閲覧・編集用)でエクスポート/インポート可能とする | IndexedDBはブラウザ削除で消失するため、ファイルベースの退避手段が必須 |
| ライブラリ利用可 | npm経由のライブラリを許可(例: SheetJS、encoding.js、Dexie.js、Rechartsなど)。CDN依存は避けビルドに含める | Vercel運用前提のためVanilla JS制約は不要 |

### 3.2 データ消失リスクへの対策(重要)

IndexedDB永続化には「ブラウザのサイトデータ削除で全データ消失」というリスクが伴う。対策として以下を必須要件とする。

- 起動時に最終バックアップ日時を表示し、一定期間(既定: 7日)を超過していたら警告
- ワンクリックで完全バックアップ(JSON)をダウンロードできる導線を全画面に常設
- `navigator.storage.persist()` による永続化リクエスト
- 端末間(PC↔スマホ)のデータ移行・同期は**JSONバックアップの手動エクスポート/インポート**で行う。自動同期機構は設けない(C6決定)

## 4. 機能要件

### 4.1 画面構成(5画面)

| # | 画面 | 概要 |
|---|---|---|
| S1 | 取引入力・インポート | 手動入力フォーム+SBI証券CSVインポート(重複検出付き) |
| S2 | ジャーナル | 取引単位/日単位の振り返り記述。感情タグ、ルール遵守評価、長文メモ |
| S3 | ルール仕様エディタ | 売買ルールの構造化記述とバージョン管理、Markdownエクスポート |
| S4 | 分析ダッシュボード | ルール別・銘柄別・タグ別の成績集計、損益分布、資産推移 |
| S5 | ポートフォリオ&リバランス | 現在配分の可視化、目標配分との乖離、必要売買数量の算出 |

### 4.2 F1: トレードジャーナル(S1, S2, S4)

- 取引記録のCRUD。SBI証券CSVインポート時は6.2節の重複判定キーで重複を検出し、衝突時はユーザーが取込可否を判断する
- 売却取引に対し、対応する買付をFIFO(先入先出)で自動マッチングし実現損益を算出する。手動での紐付け修正も可能とする
- ジャーナルエントリは取引単位・日単位の両方で作成可能。長文(数千字)の記述と全文検索に対応
- 感情タグ(例: 焦り/確信/迷い/退屈)と自由タグを付与可能。タグマスタはユーザー編集可
- 各取引に「適用ルール(バージョン付き)」と「ルール遵守評価(遵守/一部逸脱/逸脱)」を記録
- 集計軸: ルール別勝率・期待値、遵守/逸脱別の成績比較、感情タグ別成績、銘柄別・月別損益

### 4.3 F2: 売買ルール仕様化(S3)

- ルールは以下の構造化セクションで記述する:
  - 概要(戦略の狙い、対象市場・商品)
  - エントリー条件(箇条書き、AND/OR明示)
  - イグジット条件(利確・損切り・時間切れ)
  - 除外条件(手を出さない状況)
  - 資金管理(1トレードあたり最大リスク、ポジションサイズ算出式)
- **バージョン管理**: ルール改訂時は新バージョンとして保存し、旧バージョンは不変(イミュータブル)。改訂理由の記述を必須とする。取引記録からの参照は「ルールID+バージョン番号」で行う
- Markdownエクスポート: ルール1件を1ファイルのMarkdownとして出力(Git管理を想定したファイル名規約: `rule-{slug}-v{version}.md`)
- 分析ダッシュボードから「このルールの成績」へ相互リンク

### 4.4 F3: ポートフォリオ・リバランス(S5)

- 保有ポジションは取引記録から自動算出(取得単価は移動平均)。SBI証券のポートフォリオCSVインポートによる評価額・現在値の反映にも対応
- 現在値の更新手段は「ポートフォリオCSV再インポート」または「手動入力」とする(無料運用のため株価APIは使わない)
- **セクター分類(C3決定)**: 銘柄セクターマスタ(Sector)を設け、各銘柄にセクターをタグ付けする。マスタはアプリ内で手動編集できるほか、CSVインポート/エクスポートに対応し、手動作成したマスタファイルの取り込みを可能とする
- 目標配分はアセットクラス階層(例: 日本株/米国株/投信・ETF/現金)に加え、**セクターレベルの配分指標**を定義・可視化できる(セクター別の現在比率と目標比率の乖離を表示)
- リバランス計算: 目標配分との乖離額・乖離率を算出し、必要売買数量(単元株・口数を考慮)を提示。「売却なし・買付のみ(ノーセルリバランス)」モードを備える
- **通貨(C7決定)**: 米国株はUSD建てで完結管理する(損益・集計もUSDで閉じる)。JPY換算は全体俯瞰の表示目的のみとし、精度は重視しない。換算レートは受渡金額に円貨情報が得られる場合は逆算し、得られない場合は手動入力の概算レートを用いる
- NISA枠管理: 成長投資枠・つみたて投資枠の年間利用額と残枠を手入力ベースで管理し、リバランス提案時に「NISA枠内で買付可能か」を表示

## 5. データモデル(論理設計)

### 5.1 エンティティ一覧

| エンティティ | 主キー | 概要 |
|---|---|---|
| Security(銘柄) | security_id | 銘柄マスタ。コード、名称、市場区分、商品種別、通貨、単元株数、セクター(sector_id、任意) |
| Sector(セクター) | sector_id | 銘柄セクターマスタ。名称、表示順。手動編集+CSVインポート/エクスポート対応 |
| Trade(取引) | trade_id | 約定単位の記録。売買区分、数量、単価、手数料、税、口座区分(特定/NISA成長/NISAつみたて/旧NISA) |
| TradeMatch(損益マッチング) | match_id | 売却Tradeと買付Tradeの対応(FIFO自動+手動修正)。実現損益を保持 |
| JournalEntry(ジャーナル) | journal_id | 振り返り記述。対象(trade_id または 日付)、本文。※遵守評価はTradeRuleLinkに一元化(重複保持しない) |
| Tag / JournalTag | tag_id | 感情タグ・自由タグのマスタと関連 |
| Rule(売買ルール) | rule_id | ルールの識別子とメタ情報(名称、状態: 有効/廃止) |
| RuleVersion(ルール版) | rule_id + version | 構造化セクション本文、改訂理由、作成日時。イミュータブル |
| TradeRuleLink | trade_id | 取引と RuleVersion の紐付け+遵守評価 |
| PriceSnapshot(価格) | security_id + 日時 | 現在値・評価額の記録(CSVインポートまたは手動) |
| FxRate(為替) | 通貨ペア + 日時 | 手動入力レート |
| TargetAllocation(目標配分) | allocation_id | 階層構造の目標比率(合計100%検証) |
| NisaUsage(NISA枠) | 年 + 枠種別 | 年間利用額の手入力管理 |
| AppMeta | key | 最終バックアップ日時等の運用メタ情報 |

### 5.2 主要リレーション

```
Rule 1 ─ * RuleVersion 1 ─ * TradeRuleLink * ─ 1 Trade * ─ 1 Security * ─ 1 Sector(任意)
Trade(売) 1 ─ * TradeMatch * ─ 1 Trade(買)
JournalEntry * ─ 1 Trade(nullable。日単位エントリの場合はnull+対象日付)
Security 1 ─ * PriceSnapshot
```

### 5.3 設計上の決定事項

- **損益マッチングはFIFO(先入先出)で確定**(C2決定)。移動平均法との併存は行わない(取得単価表示は保有計算上の移動平均を参考値として持つが、実現損益の計算はFIFOに一本化)
- 金額は円は整数(円)、USDは小数第2位までの固定小数(内部は最小単位の整数)で保持し、浮動小数点誤差を排除する
- 投資信託は「口数」を数量として扱い、株式と同一のTradeエンティティで表現する(商品種別で分岐)
- 選択肢系フィールド(売買区分、口座区分、商品種別等)はコード値とラベルを分離して定義する

## 6. 外部データ連携(SBI証券CSV)

> 2026-07-06に実ファイル3種(約定履歴照会CSV・ポートフォリオCSV・米国株式約定履歴CSV)を解析し、本章を確定仕様とした。

### 6.1 インポート対象

| ファイル | 用途 | 確認状況 |
|---|---|---|
| 約定履歴照会CSV(`SaveFile_*.csv`) | Trade取り込み。国内株式・国内ETF・投資信託の取引が**1ファイルに混在** | **確定**(実ファイル確認済み) |
| ポートフォリオCSV(`ポートフォリオ_YYYYMMDD.csv`) | PriceSnapshot取り込み+保有数量の突合検証 | **確定**(実ファイル確認済み) |
| 米国株式 約定履歴CSV(`PaymentRecords_YYYYMMDD.csv`) | Trade取り込み(USD建て) | **確定**(実ファイル確認済み) |

### 6.2 約定履歴照会CSV(確定仕様)

**ファイル形式**
- 文字コード: Shift-JIS(CP932)、BOMなし、改行LF
- 構造: 先頭にプリアンブル(タイトル「約定履歴照会」、抽出条件行、注記、空行)。その後にヘッダー行、明細行群、末尾に空行
- ヘッダー行は「約定日」で始まる行として検出する(**行位置固定に依存しない**)

**列定義(14列)**

| 列名 | 変換ルール |
|---|---|
| 約定日 / 受渡日 | `YYYY/MM/DD` |
| 銘柄 | 全角英数・全角スペース(U+3000)を含む。照合用にNFKC正規化キーを併用 |
| 銘柄コード | 株式・ETFは4桁英数字(例: `1489`、`314A`)。**投資信託は空** |
| 市場 | 観測値: `東証` / `東証(外)` / `--` / 空 |
| 取引 | 観測値: `株式現物買` / `株式現物売` / `投信金額買付` / `投信金額解約` / `分配金再投資` |
| 期限 | 現物では `--` |
| 預り | 観測値: `NISA(成)` / `NISA(つ)` / `特定` / `旧NISA`。**前後に半角スペースを含むためトリム必須** |
| 課税 | 観測値: `非課税` / `申告` / `課税` / `--` |
| 約定数量 | 株式・ETF: 株数。投信: 口数 |
| 約定単価 | 株式・ETF: 円(小数あり得る。例 `2180.5`)。投信: **1万口あたり基準価額(円)** |
| 手数料/諸経費等 / 税額 | 観測値は `--`(ゼロ扱い) |
| 受渡金額/決済損益 | 円(整数)。投信は「口数×基準価額÷10,000」と端数レベルで一致(**受渡金額を正とする**) |

**取り込みルール**
- `--` および空文字はnull(数値項目はゼロ)として扱う
- 商品種別の判定: 「取引」列が `投信金額買付` / `投信金額解約` / `分配金再投資` → 投資信託。それ以外 → 株式/ETF(ETFは株式現物として記録されるため、Security側の商品種別で区別する)
- `分配金再投資` は買付として取り込む(数量・単価を持つ通常の買付と同様に扱い、取得原価に算入)
- 重複判定キー: 約定日+銘柄識別子(コード、投信は正規化名称)+取引区分+約定数量+約定単価。同一キーの複数約定が正当に存在し得るため、**インポート時に同一キー衝突を検出したら自動スキップせず、ユーザーに判断を求める**

### 6.3 ポートフォリオCSV(確定仕様)

**ファイル形式**
- 文字コード: Shift-JIS(CP932)、BOMなし。**各行末尾に余分なカンマ(空列)がある**
- 構造: マルチセクション形式
  1. プリアンブル(タイトル「ポートフォリオ一覧」、表示条件、件数等)
  2. 繰り返しブロック: セクションタイトル行 → 明細ヘッダー行 → 明細行群 → 合計ブロック(タイトル行+合計ヘッダー行+合計値1行)
  3. 末尾に「総合計」ブロック
- セクションタイトル(例: `株式(現物/NISA預り(成長投資枠))`、`投資信託(金額/NISA預り(つみたて投資枠))`)を**パースして商品種別と口座区分を取得**する

**明細列(7列)**: 銘柄(コード) / 買付日 / 数量 / 取得単価 / 現在値 / 損益 / 損益(%)

| 項目 | 変換ルール |
|---|---|
| 銘柄(コード) | 株式・ETF: `7267 ホンダ` 形式(コード+半角スペース+略称)。先頭トークンをコードとして分離。投信: ファンド名のみでコードなし |
| 投信の名称照合 | 略称・表記ゆれに備えNFKC正規化キーで銘柄マスタと照合。**照合失敗時はユーザーが対話的に対応付け、エイリアスとして保存**(次回から自動照合) |
| 買付日 | `----/--/--` の場合あり(null扱い) |
| 取得単価・現在値 | 株式・ETF: 円。投信: 1万口あたり基準価額(円) |
| 損益 | 符号付き・小数あり(例 `+11385.27`) |

**取り込みルール**
- 明細行のみをPriceSnapshotに反映する。合計ブロックは取り込まず、明細の再計算値との**突合検証**に利用する(不一致時は警告表示)
- ポートフォリオCSVの保有数量と、取引記録から算出した保有数量が不一致の場合は差異レポートを表示する(取引履歴の取り込み漏れ検出に利用)

### 6.4 米国株式 約定履歴CSV(確定仕様)

**ファイル形式**
- 文字コード: Shift-JIS(CP932)、BOMなし、改行LF。全フィールドがダブルクォート囲み
- 構造: 先頭にプリアンブル(タイトル「約定履歴」、抽出条件行、空行)。その後にヘッダー行、明細行群
- ヘッダー行は「国内約定日」で始まる行として検出する(**行位置固定に依存しない**)

**列定義(12列)**

| 列名 | 変換ルール |
|---|---|
| 国内約定日 / 国内受渡日 | `YYYY/MM/DD`(日本時間基準の日付) |
| 銘柄 | 日本語名称(カナ表記)。全角スペース(U+3000)を含む場合あり |
| 銘柄コード | ティッカーシンボル(例: `T`、`SPYD`)。国内4桁コードと形式が異なるため、Securityの識別子は市場込みで一意化する |
| 市場 | 観測値: `NYSE` / `NASDAQ` / `NYSE Arca` |
| 商品区分 | 観測値: `米国株式`(米国ETFも同区分に含まれる) |
| 注文種別 | 観測値: `指値` / `成行`(取り込み時は参考情報として保持) |
| 取引 | 観測値: `現買` / `現売`。**国内版(株式現物買/売)と語彙が異なる**ため、共通の売買区分コードへマッピングする |
| 預り区分 | 観測値: `NISA` / `特定` / `旧NISA`。**国内版と異なり前後スペースなし、成長/つみたての別なし**(`NISA`→NISA成長として扱う) |
| 約定数量 | 株数 |
| 約定単価 | `20.5100USD` 形式(**数値+通貨コードサフィックス**、小数第4位まで)。サフィックスをパースして通貨を判定する |
| 受渡金額/決済損益 | `104.49USD` 形式(小数第2位まで)。売却時は手数料等控除後の**手取り額**(数量×単価と一致しない。例: 4株×26.25=105.00に対し104.49) |

**取り込みルール**
- 通貨: 明細はUSD建て。円貨換算レートは本CSVに含まれないため、JPY換算はFxRate(手動入力)による(C7参照)
- 手数料: 独立した列がないため、「|数量×約定単価 − 受渡金額|」を実効コスト(手数料・諸費用相当)として導出し、参考値として保持する
- 重複判定キー: 6.2節と同一方式(約定日+ティッカー+市場+取引区分+数量+単価)
- ETF判定: 商品区分では株式とETFを区別できないため、国内版と同様にSecurity側の商品種別で管理する

### 6.5 インポート共通仕様

- 文字コード自動判定: Shift-JIS / BOM付きUTF-8 / UTF-8 の3種を判定して読み込む(実ファイルはShift-JISと確定したが、Excel等で開いて再保存されたファイルの受け入れも考慮する)
- ヘッダー行の列名でファイル種別を自動識別し、不一致時は明示的なエラーメッセージを表示する(黙って誤マッピングしない)
- **列マッピング定義は外部化**(設定オブジェクトとして分離)し、SBI側のフォーマット変更時にマッピングのみ修正できる構造とする
- インポート結果サマリー(取込件数/重複スキップ件数/エラー件数と行番号)を必ず表示する

## 7. エクスポート仕様

- 完全バックアップ: 全エンティティをJSON 1ファイルで出力(スキーマバージョン番号を含め、将来のマイグレーションに備える)
- Excel/CSVエクスポート: 取引一覧・ジャーナル・保有一覧をBOM付きUTF-8で出力(Excel往復編集対応)
- ルール仕様: Markdownファイル出力(1ルール1ファイル)

## 8. 非機能要件

- 性能: 取引5,000件・ジャーナル3,000件規模で、一覧表示・全文検索が体感1秒以内

- 可用性: **PWA化をv1に含める**(C5決定)。オフラインキャッシュ(Service Worker)とホーム画面追加に対応し、オフラインでも閲覧・入力可能とする
